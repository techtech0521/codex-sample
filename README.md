# codex-sample

## MVP 技術方針

このプロジェクトでは、MVP（Minimum Viable Product）を素早く検証し、後から機能拡張しやすい構成を優先します。

### 1. アプリ形態

**Webアプリ** として開発します。

- ブラウザから利用でき、PC・スマートフォンの両方でアクセスしやすい
- ストア申請が不要で、MVPを素早く公開・改善できる
- 将来的にモバイルアプリ化する場合も、Web版で検証した仕様を流用しやすい

### 2. フロントエンド

**React / Next.js** を採用候補にします。

- 画面開発、ルーティング、API連携を一体で進めやすい
- MVPから本番運用まで拡張しやすい
- UIライブラリや周辺エコシステムが豊富

### 3. バックエンド

**Next.js API Routes** を採用候補にします。

- フロントエンドと同じリポジトリ内でAPIを管理できる
- 小規模なMVPでは構成をシンプルに保てる
- 必要に応じて、将来的にNode.js / ExpressやPython / FastAPIへ分離しやすい

### 4. データベース

**PostgreSQL** を採用候補にします。

- リレーショナルデータを扱いやすく、将来の機能追加に対応しやすい
- Supabaseや各種クラウドDBへ移行・接続しやすい
- MVP後にデータ量や機能が増えても運用しやすい

### 5. MVPでの認証方針

**最初は認証なし** で開始します。

- 初期開発では、主要機能の検証を優先する
- ログインやユーザー管理による実装コストを後回しにする
- 利用シーンや保存データの要件が固まった段階で、ログイン機能やユーザー別ログ管理を追加する

## 採用方針まとめ

| 項目 | 方針 |
| --- | --- |
| アプリ形態 | Webアプリ |
| フロントエンド | React / Next.js |
| バックエンド | Next.js API Routes |
| データベース | PostgreSQL |
| MVP認証 | 最初は認証なし |

## 学習ログデータモデル

MVPでは、1件の学習記録を `LearningLog` として扱います。最初は認証なしで開始する方針のため、`userId` は将来拡張項目として任意にします。

### 基本フィールド

| フィールド | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| `id` | `string` / UUID | 必須 | 学習ログの一意なID |
| `date` | `date` | 必須 | 学習日。日付単位で検索・集計しやすい形式にする |
| `title` | `string` | 必須 | 学習内容のタイトル |
| `category` | `string` | 必須 | 学習カテゴリ。例: 英語、プログラミング、資格、読書 |
| `durationMinutes` | `number` / integer | 必須 | 学習時間。分単位で保存し、時間表示は画面側で変換する |
| `memo` | `string` / text | 任意 | 学習メモ。振り返りや補足内容を保存する |
| `createdAt` | `datetime` | 必須 | 作成日時 |
| `updatedAt` | `datetime` | 必須 | 更新日時 |

### 将来拡張フィールド

| フィールド | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| `materialName` | `string` | 任意 | 教材名。書籍名、講座名、動画名など |
| `difficulty` | `string` / enum | 任意 | 難易度。例: easy、normal、hard |
| `mood` | `string` / enum | 任意 | 集中度や気分。例: focused、normal、tired |
| `tags` | `string[]` | 任意 | タグ。横断的な検索や分類に利用する |
| `userId` | `string` / UUID | 任意 | ユーザーID。認証機能を追加した後に利用する |

### TypeScript型定義案

```ts
export type LearningLogDifficulty = 'easy' | 'normal' | 'hard';

export type LearningLogMood = 'focused' | 'normal' | 'tired';

export type LearningLog = {
  id: string;
  date: string;
  title: string;
  category: string;
  durationMinutes: number;
  memo?: string;
  createdAt: string;
  updatedAt: string;
  materialName?: string;
  difficulty?: LearningLogDifficulty;
  mood?: LearningLogMood;
  tags?: string[];
  userId?: string;
};
```

### PostgreSQLテーブル設計案

```sql
CREATE TABLE learning_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  date DATE NOT NULL,
  title VARCHAR(255) NOT NULL,
  category VARCHAR(100) NOT NULL,
  duration_minutes INTEGER NOT NULL CHECK (duration_minutes >= 0),
  memo TEXT,
  material_name VARCHAR(255),
  difficulty VARCHAR(50),
  mood VARCHAR(50),
  tags TEXT[] DEFAULT '{}',
  user_id UUID,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_learning_logs_date ON learning_logs (date);
CREATE INDEX idx_learning_logs_category ON learning_logs (category);
CREATE INDEX idx_learning_logs_tags ON learning_logs USING GIN (tags);
```

### 設計メモ

- `durationMinutes` は数値として保存し、合計学習時間やカテゴリ別集計をしやすくします。
- `date` は学習した日、`createdAt` / `updatedAt` はレコード管理用の日時として役割を分けます。
- MVPでは `category` を文字列で始め、カテゴリ管理が必要になった段階で別テーブル化を検討します。
- `tags` は複数条件で検索できるよう配列で持たせます。タグの表記ゆれが課題になった場合は別テーブル化します。
- 認証追加後は `userId` にインデックスを追加し、ユーザー別に学習ログを取得できるようにします。

## 画面設計（初期実装）

最初の実装では、主要導線を素早く検証するために **ホーム画面**、**学習ログ一覧画面**、**学習ログ登録画面** の3画面から開始します。編集画面と集計画面は、初期3画面のデータ登録・閲覧体験が固まった後に追加します。

### 共通レイアウト

- ヘッダーにはアプリ名「Study Log」と主要ナビゲーションを配置します。
- 主要ナビゲーションは「ホーム」「ログ一覧」「ログ登録」「集計」で構成します。
- MVPでは認証なしのため、ユーザー名・ログアウト導線は表示しません。
- PCでは最大幅を設けた中央寄せレイアウト、スマートフォンでは1カラム表示にします。

### 1. ホーム画面

#### 目的

学習状況の概要をすぐ確認し、最近のログ確認や新規登録へ移動できる入口にします。

#### 表示項目

| エリア | 内容 |
| --- | --- |
| 今日の学習時間 | 当日の日付に一致するログの `durationMinutes` 合計を「時間 分」形式で表示 |
| 今週の学習時間 | 週の開始日から今日までのログの `durationMinutes` 合計を表示 |
| 最近の学習ログ | 日付降順で直近5件のログを表示 |
| アクション | 「学習ログを登録」ボタン、「すべてのログを見る」リンク |

#### 最近の学習ログカード

- 日付
- タイトル
- カテゴリ
- 学習時間
- メモの先頭数十文字（任意）

#### 空状態

学習ログが未登録の場合は、「まだ学習ログがありません。最初のログを登録しましょう。」と表示し、登録画面へのボタンを出します。

### 2. 学習ログ一覧画面

#### 目的

登録済みの学習ログを日付順で確認し、カテゴリやキーワードで目的のログを探せるようにします。

#### 表示項目

| エリア | 内容 |
| --- | --- |
| 検索フォーム | キーワード入力、カテゴリ選択、検索/クリアボタン |
| ログ一覧 | 日付降順でログを表示 |
| 件数表示 | 絞り込み後の表示件数を表示 |

#### 絞り込み条件

- カテゴリ: `category` の完全一致で絞り込みます。
- キーワード: `title` と `memo` を対象に部分一致で検索します。
- 条件未指定時は、全ログを日付降順で表示します。

#### ログ一覧の表示形式

| 項目 | 表示内容 |
| --- | --- |
| 日付 | `date` |
| タイトル | `title` |
| カテゴリ | `category` |
| 学習時間 | `durationMinutes` を「時間 分」形式に変換 |
| メモ | `memo` の概要 |

#### 空状態

検索条件に一致するログがない場合は、「条件に一致する学習ログがありません。」と表示し、絞り込み条件のクリアボタンを出します。

### 3. 学習ログ登録画面

#### 目的

学習内容を迷わず記録できる、最小限の入力フォームを提供します。

#### 入力項目

| 項目 | 入力形式 | 必須 | バリデーション |
| --- | --- | --- | --- |
| 日付 | date input | 必須 | 未入力不可。初期値は今日の日付 |
| タイトル | text input | 必須 | 1〜255文字 |
| カテゴリ | select または text input | 必須 | 1〜100文字。初期候補は「英語」「プログラミング」「資格」「読書」 |
| 学習時間 | number input | 必須 | 0以上の整数。分単位で入力 |
| メモ | textarea | 任意 | 長文入力可 |

#### アクション

- 「登録する」ボタンで保存します。
- 登録成功後は、ホーム画面または一覧画面へ遷移します。
- 「キャンセル」リンクで一覧画面へ戻ります。

#### エラー表示

- 必須項目が未入力の場合は、該当項目の下にエラーメッセージを表示します。
- 学習時間が負数または整数以外の場合は、「学習時間は0以上の整数で入力してください。」と表示します。

## 将来追加する画面設計

### 4. 学習ログ編集画面

#### 目的

既存ログの内容を修正・削除できるようにします。

#### 表示・入力項目

登録画面と同じ入力項目を、既存ログの値で初期表示します。

#### アクション

- 「更新する」ボタンで修正内容を保存します。
- 「削除」ボタンで対象ログを削除します。
- 誤削除を防ぐため、削除前に確認ダイアログを表示します。
- 「キャンセル」リンクで一覧画面または詳細画面へ戻ります。

### 5. 集計画面

#### 目的

学習時間の推移やカテゴリ別の偏りを確認し、振り返りに活用できるようにします。

#### 表示項目

| 集計 | 内容 |
| --- | --- |
| 日別学習時間 | 日付ごとの `durationMinutes` 合計を表示 |
| 週別学習時間 | 週ごとの `durationMinutes` 合計を表示 |
| カテゴリ別学習時間 | カテゴリごとの `durationMinutes` 合計を表示 |

#### 表示形式

- MVPでは表形式から開始します。
- 余裕があれば棒グラフや円グラフを追加します。
- 集計期間は初期表示で直近30日、将来的に期間指定を追加します。

## 初期実装の優先順位

1. 学習ログ登録画面でログを作成できるようにする。
2. 学習ログ一覧画面で登録済みログを日付降順に表示する。
3. 一覧画面にカテゴリ絞り込みとキーワード検索を追加する。
4. ホーム画面に今日・今週の学習時間と最近の学習ログを表示する。
5. 編集画面と集計画面を後続リリースで追加する。
